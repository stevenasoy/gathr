# Gathr Performance + Security Audit

## Executive summary

Gathr's security and performance posture is **weak across the board**, with no single dimension clean. The three top risks: (1) **Supabase sessions — including long-lived refresh tokens — are persisted to `localStorage`** on the web client, so any XSS yields silent indefinite account takeover; (2) **RLS policies on `bookings` and `conversations` permit hosts/guests to mutate fields they should not** (event_date, guest count, venue_id) directly from the browser console, bypassing the API's narrower handlers; (3) **there is no dependency-vulnerability gate in CI and no input-validation layer at the API boundary**, so known advisories and malformed/malicious payloads sail through. On performance, the API has a classic **N+1 in the messages `/latest` endpoint with no cap on `ids.length`** (pool-exhaustion DoS), two-query waterfalls on every host booking/venue mutation, an effectively unbounded 500-row default on the messages list, and a non-materialized review-stats view that scans the whole reviews table per request. Frontend re-renders are rampant (un-memoized AuthContext/SavedContext, un-memoized VenueCard, per-keystroke recomputation on the Venue page) and images consistently lack dimensions/responsive sources. This is pre-launch code that needs hardening before it should accept real user data.

## Critical findings

**None.** No finding reached the "critical" severity band after adversarial verification. The four findings most likely to escalate in impact — localStorage session exposure, RLS host over-mutation, the uncapped N+1, and the missing audit gate — are all rated **high** below.

## High findings

### Security

**[H1] Supabase session (access + refresh tokens) persisted to `localStorage` — XSS-exposed** — `apps/web/src/lib/supabase.ts:11` — The web client is created with no `auth` config override, so supabase-js defaults to `persistSession: true` + `localStorage`, writing the full session JSON (including the long-lived refresh token) to a JS-readable key; a single XSS exfiltrates the refresh token for indefinite silent account takeover. **Fix:** adopt `@supabase/ssr` and store the session in an httpOnly, Secure, SameSite=Lax cookie set by a server route, or as an interim set `auth: { persistSession: false, autoRefreshToken: false }` for in-memory-only sessions; add a strict CSP to the web app.

**[H2] `bookings` host update RLS allows host to mutate non-status fields (event_date, hours, guests, note)** — `supabase/v5-schema.sql:84` — The host-update policy only verifies venue ownership; the `guard_bookings_immutable` trigger only protects `user_id`/`venue_id`/`total_php`, leaving `event_date`/`hours`/`guests`/`event_type`/`venue_name`/`note` mutable by any signed-in host via a direct `supabase.from('bookings').update(...)` from the browser console. **Fix:** add a BEFORE UPDATE trigger that raises unless only `status` changed (mirror `guard_bookings_immutable`), since RLS `WITH CHECK` cannot compare OLD vs NEW.

**[H3] No `npm audit` / dependency-vulnerability gate in CI** — `.github/workflows/ci.yml:1` — The four CI jobs (lint/build/test/docker/e2e) run no `npm audit`, `audit-ci`, Snyk, or Dependabot step, so critical/high advisories (including the vitest GHSA-5xrq-8626-4rwp critical found in this audit) land on `main` undetected. **Fix:** add an `audit` job running `npm audit --audit-level=high` (or `audit-ci --moderate`) gated on push/PR to main, and enable Dependabot security updates (`.github/dependabot.yml`, npm, weekly).

### Performance

**[H4] N+1 queries in `POST /api/messages/latest` — one DB round-trip per thread ID, no cap on `ids.length`** — `apps/api/src/routes/messages.ts:88` — The endpoint loops over the client-supplied `ids` array firing one Supabase query per thread (parallelized with `Promise.all`, so max-of-N latency but N concurrent connections/requests), with no upper bound — a client passing 1000 IDs triggers 1000 parallel queries that can exhaust the Postgres connection pool and starve other requests. **Fix:** replace with a single query using `DISTINCT ON (thread_id)` (RPC/view) ordered by `(col, created_at desc)`, and cap `ids.length` (e.g. max 200) with a 400-reject for larger batches.

## Medium findings

### Performance — Frontend

**[M1] AuthContext value object and methods not memoized** — `apps/web/src/context/AuthContext.tsx` — Every AuthProvider render builds a new value object and recreates `signUp`/`signIn`/`signOut`, re-rendering all `useAuth` consumers even when `user` is unchanged. **Fix:** wrap methods in `useCallback` and the value in `useMemo` keyed on `[user, loading, displayName, signUp, signIn, signOut, configured]`.

**[M2] VenueCard not memoized and consumes SavedContext** — `apps/web/src/components/VenueCard.tsx` — Toggling save on one card re-renders every visible card; every keystroke in Search re-renders all result cards. **Fix:** wrap VenueCard in `React.memo`; split SavedContext so `isSaved`/`toggle` are narrower, or pass `saved`/`onToggle` as props from a single subscribing parent.

**[M3] Venue detail page recomputes derived data on every keystroke** — `apps/web/src/pages/Venue.tsx` — `typeLabels`, `similar`, `myOther`, and `todayYMD()` recompute on every form-field render. **Fix:** wrap in `useMemo` with deps `[venues, venue, user]`; compute `todayStr` once at module scope.

**[M4] VenueCard images lack width/height, srcset/sizes; fixed 900px source** — `apps/web/src/components/VenueCard.tsx` — CLS on grid load; mobile downloads 900px-wide images. **Fix:** add `width`/`height` (or `aspect-ratio`) and a responsive `srcset` (400/800/1200) + `sizes` using the existing `img()` width param.

**[M5] Search renders the entire result set with no virtualization or cap** — `apps/web/src/pages/Search.tsx` — Unbounded DOM + simultaneous image loads on large catalogs. **Fix:** cap at ~24 with "Load more", or virtualize with `@tanstack/react-virtual`.

**[M6] NotificationsContext realtime channel tears down/re-subscribes on every refresh** — `apps/web/src/context/NotificationsContext.tsx` — Effect depends on the `venueIds` array identity, which changes every `refresh()`. **Fix:** depend on `venueIds.join(',')` so the channel only re-subscribes when the actual set changes.

### Performance — Backend

**[M7] `loadBookingIfParty` runs a 2-query waterfall (booking → venue) on every host booking access** — `apps/api/src/routes/bookings.ts:27` — Every host-side GET/PATCH/DELETE pays 2 sequential DB round-trips for authz. **Fix:** use a single nested select `.select('id,user_id,venue_id,status,venues!venue_id(owner_id)')` and read `data.venues?.owner_id`.

**[M8] `GET /api/venues/my` has no pagination — returns every owned venue in one shot** — `apps/api/src/routes/venues.ts:54` — Full `VENUE_COLUMNS` (incl. `image_urls`, `blurb`) with no limit/offset. **Fix:** apply the same limit/offset as GET / (default 50, cap 100) with a total count; project a lighter column set for lists.

**[M9] `GET /api/reviews/stats` fallback scans every review row in-memory; no caching on either path** — `apps/api/src/routes/reviews.ts:44` — Fallback pulls the entire reviews table into Node and aggregates in JS; even the view path is recomputed per request with no cache. **Fix:** cache the response (in-memory LRU, 30–60s TTL, invalidate on review insert); on fallback, aggregate DB-side via RPC — never stream all reviews into Node.

**[M10] `GET /api/messages` default limit 500, effectively unbounded, with no upper-bound clamp** — `apps/api/src/routes/messages.ts:25` — `select('*')` for up to 500 full `body` rows on a hot path; a client can pass `limit=100000`. **Fix:** default 50–100, clamp client limit to a max (e.g. `Math.min(...||50, 200)`), select only needed columns, paginate via keyset/cursor on `created_at`.

**[M11] Every authenticated request makes a network round-trip to Supabase Auth via `auth.getUser`** — `apps/api/src/middleware/auth.ts:21` — A per-request auth-service hop on the critical path of every route; the JWT is signed and verifiable locally. **Fix:** verify the JWT locally with the Supabase JWT secret (`jose` `jwtVerify`) to populate `r.user` without a network call; fall back to `auth.getUser()` only for edge cases; optionally add a short-TTL in-memory cache keyed by jti.

### Performance — Database

**[M12] v6 adds duplicate message indexes already created in v2/v3** — `supabase/v6-performance.sql:25` — `messages_booking_created_idx`/`messages_conversation_created_idx` duplicate v2/v3's `(fk, created_at)` indexes (btree is reversible), so every INSERT maintains 4 indexes instead of 2. **Fix:** drop the redundant pair (either set); keep one index per thread FK.

**[M13] v6 `venue_review_stats` is a plain VIEW, not materialized — full reviews scan per query** — `supabase/v6-performance.sql:10` — A non-materialized view expands to `GROUP BY` over the entire reviews table on every SELECT; v6's perf claim is false. **Fix:** make it a materialized view with a unique index on `venue_id`, refreshed concurrently on review insert/update/delete via trigger.

**[M14] No time-based indexes on bookings/messages/reviews/conversations** — `supabase/v5-schema.sql:270` — `created_at` only appears as a trailing composite column tied to a parent FK; time-range queries, analytics, and cleanup jobs seq scan. **Fix:** add `*_created_idx` (desc) on each of the four growth tables; consider range partitioning at scale.

**[M15] v6-performance.sql left all real performance gaps open** — `supabase/v6-performance.sql:1` — Only added one non-materialized view + 2 redundant message indexes; no GIN on types/amenities, no city/area index, no `(user_id, created_at)` composites, no event_date index, no fix for low-selectivity status indexes. **Fix:** address in a v7 migration (see specific findings M12–M14, L13–L17).

### Security — Auth

**[M16] No password reset / forgot-password flow; no reset-token expiry configured** — `apps/web/src/pages/SignIn.tsx:52` — Zero `resetPasswordForEmail` calls in the repo; no "Forgot password?" link; no app-level control over reset-token lifetime. **Fix:** add `/forgot-password` → `resetPasswordForEmail(email, { redirectTo: '/reset-password' })` and `/reset-password` → `updateUser({ password })`; explicitly set and document token expiry (e.g. 1 hour) in Supabase Auth settings and `docs/ops.md`.

**[M17] Login/signup bypasses the Express API entirely — no app-level auth rate limiting or lockout** — `apps/web/src/context/AuthContext.tsx:69` — Auth runs direct browser→Supabase; the API's rate limiters never see credential traffic; brute-force protection depends on unconfigured Supabase platform defaults. **Fix:** proxy sign-in/sign-up through the API with a strict per-IP+email limiter (e.g. 5/15min) calling Supabase server-side, or explicitly verify/document Supabase Auth limits and add client-side backoff + lockout.

### Security — Input validation

**[M18] No schema-validation library at the API boundary; all write endpoints rely on ad-hoc presence checks** — `apps/api/package.json:1` — No zod/yup/joi; every POST/PUT/PATCH validates only `if (!field)` + a `pick()` allowlist — no type/range/enum/length validation on any write. **Fix:** add `zod`, define a schema per write endpoint (`safeParse` before Supabase), e.g. `rating: z.number().int().min(1).max(5)`, `total_php: z.number().min(0)`, `status: z.enum([...])`, length caps on `body`/`note`/`author_name`.

### Security — Config

**[M19] `.gitignore` covers `.env` but not key/cert files (`*.pem`, `*.key`, `*.p12`, `*.pfx`)** — `.gitignore:7` — A stray TLS key or service-account JSON committed by accident is not auto-ignored. **Fix:** add a key-material block: `*.pem`, `*.key`, `*.crt`, `*.p12`, `*.pfx`, `*.cer`, `*service-account*.json`, `.env.local`, `.env.*.local`.

**[M20] Production error handler leaks raw `err.message` to clients** — `apps/api/src/index.ts:104` — Only `err.stack` is suppressed; raw driver messages (table/column/constraint names) leak for any non-23505 error. **Fix:** for 5xx in production, return a generic message; map additional Postgres codes (23503→409/422, 22P02→400) to safe messages.

### Security — RLS

**[M21] `reviews` public read exposes `reviews.user_id` (auth.users uuid) to anonymous users** — `supabase/v4-schema.sql:52` — `for select using (true)` exposes every column including the stable cross-site reviewer uuid; only `author_name` is needed publicly. **Fix:** create a `reviews_public` view excluding `user_id` and grant anon/authenticated select on it; revoke broad table select.

**[M22] `venues` public read exposes `venues.owner_id` (auth.users uuid) to anonymous users** — `supabase/v2-schema.sql:17` — `status='live'` is correctly gated but all columns (incl. host uuid) are exposed; `host_name` is the display field. **Fix:** create a `venues_live` view exposing only listing UI columns and grant anon/authenticated select; restrict table-level public select.

**[M23] `conversations` update guest lets a guest change `venue_id`/`venue_name` on their own row** — `supabase/v5-schema.sql:232` — No immutability trigger on conversations (unlike bookings/reviews); a guest can retroactively re-attribute a thread. **Fix:** add a `guard_conversations_immutable()` BEFORE UPDATE trigger raising if `venue_id`/`venue_name`/`guest_id` change (mirror `guard_bookings_immutable`).

### Security — Dependencies

**[M24] vite ≤6.4.2 high (GHSA-fx2h-pf6j-xcff) nested under vitest** — `apps/web/package.json:34` — The vulnerable vite@5.4.21 is nested under vitest@2.1.9 (dev/test-only, low reachability — gathr's actual dev server is top-level vite@6.4.3, fixed). **Fix:** upgrade vitest (see L30) to pull a fixed vite transitively; confirm via `npm ls vite` no nested copy remains ≤6.4.2.

## Cross-cutting themes

- **No input-validation layer.** Every write endpoint relies on hand-rolled presence checks + `pick()` allowlists (M18, L21–L24). Type/range/enum/length validation is entirely absent — the API trusts DB constraints that may not exist. This is the single most systemic security gap.
- **RLS is the only gate, and it leaks.** The web client holds the anon key + user JWT and talks to Supabase directly; RLS is therefore the real authorization boundary, yet it has column-exposure leaks (M21, M22), an over-broad host mutation policy (H2), a missing guest UPDATE policy (L17), no conversations immutability trigger (M23), and a security_definer view (L28). The API layer's narrower handlers are bypassable from the browser console.
- **Defense-in-depth inconsistency.** Some mutating routes (venues PUT/DELETE, bookings PATCH/DELETE) do explicit server-side ownership checks; others (messages, reviews, conversations inserts, list endpoints with `venue_ids`) trust RLS alone (L19, L20). A future service-role swap or RLS regression turns these into broken access control.
- **Unbounded/uncapped data paths everywhere.** N+1 with no `ids` cap (H4), 500-row default messages list with no limit clamp (M10), unpaged `/venues/my` (M8), full-table in-memory review aggregation (M9), unbounded Search DOM (M5), `select('*')` on all lists (L7), unclamped `limit`/`offset` (L24). The codebase consistently fails to bound worst-case load.
- **Frontend re-render hygiene missing.** Context values are not memoized (M1, M6), list items are not memoized and consume whole-context providers (M2), and expensive derived data recomputes per keystroke (M3, L4, L5). React.memo/useMemo/useCallback are largely unused. Images consistently lack dimensions and responsive sources (M4, L1, L2), creating CLS and over-serving.

## Measurement gaps (performance)

Per the performance-optimization skill, before optimizing further the following measurement instrumentation is missing and should be added — the findings above are static-analysis-based, not measured:

- **No Lighthouse CI / Core Web Vitals pipeline.** No Lighthouse CI config in `.github/workflows/` or the repo; Vitals (LCP/CLS/INP) are not measured on any deploy, so the image/dimension (M4, L1, L2), font-blocking (L6), and render-cost (M1–M3) findings are not quantified against budgets.
- **No Real User Monitoring (RUM).** No RUM SDK (web-vitals, Sentry, SpeedCurve, etc.) wired into the SPA, so field p75 LCP/CLS/INP and route-level TTFB are unobserved — the N+1 (H4) and auth-hop (M11) latency only manifests in real users' p99.
- **No bundle budget / bundle analysis.** No `vite-bundle-visualizer`, `size-limit`, or `bundlewatch` step in CI; the static-import finding (L3) and lazy-route gaps are invisible without a bundle report. No budget gates prevent regressions.
